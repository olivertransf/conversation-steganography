      Follow one message all the way through

      Same toy numbers reused below. Values are simplified for teaching; real runs use full AES-SIV tags and longer bitstreams.

      1. You type meet at noon Plaintext stays on your laptop.
      2. Pack mode=1 · packed ≈ shorter bytes Tool tries several compressors; keep the smallest.
      3. Seal tag(16) || ciphertext AES-SIV with AAD binding conversation, sender, chain hash, pack mode.
      4. Chunk wire0 = [0,1,sealedLen,len,P0] If the sealed blob is big, you get wire0, wire1, … each becomes one bubble.
      5. Encode bits(wire) choose tokens Model says top tokens; secret bits pick among them → cover text.
      6. WhatsApp shows yeah traffic was light for once… No ciphertext visible. Friend copies the exact text.
      7. Decode same prompt → same candidates → recover bits Join pieces, openSIV, unpack → meet at noon.

      The core trick: bits pick the next word

      This is usually the confusing part. The ciphertext is not stuffed into the letters. It controls which of the model’s likely next tokens you emit. Your friend, with the same model and chat history, rebuilds the same candidate list and reads the bits back out from which token appeared.

      Sender

      1. Ask the model: “given this prompt, what tokens are likely next?”
      2. Keep the top N (say 32), turn scores into slice sizes on a number line.
      3. Read secret bits as a point on that line; emit the token whose slice contains it.
      4. Append that token to the cover; repeat until the wire bits are spent.

      Receiver

      1. Build the exact same prompt from the shared transcript.
      2. Ask the same model for the same top-N list.
      3. See which token actually appears in the pasted cover.
      4. That slice tells you which bits were used; append them; continue.

        Peer checklist

        Decode fails if either side drifts from the shared protocol surface.

        • Same secret phrase Agree in person. Never send the phrase over the chat you are hiding in.
        • Same conversation name Used as PBKDF2 salt and bound into authenticated data.
        • Same model + revision Token ids and fingerprints must match on both machines.
        • Same generative settings Coding mode, top-n, prompts, strict style, trials — easiest if both run setup the same way.
        • Exact cover text No autocorrect, smart quotes, or reordering. Paste parts in order.
        • Local-only models No cloud API for covers. Weights stay on the device.
